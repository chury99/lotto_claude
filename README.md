# 로또 6/45 당첨번호 분석 · 예측

동행복권 사이트에서 역대 당첨번호를 크롤링해 통계를 내고, 그 통계를 근거로 다음 회차
번호를 추천하는 도구입니다. 추천 전략이 정말로 무작위보다 나은지 스스로 검증하는
백테스트 기능을 함께 제공합니다.

> **먼저 읽어주세요.** 로또 추첨은 매 회차 독립적인 균등 무작위 시행입니다. 과거에 어떤
> 번호가 많이/적게 나왔든 다음 회차의 확률은 모든 번호가 동일하게 6/45입니다.
> 이 프로젝트에 포함된 백테스트도 그 사실을 그대로 보여줍니다(아래 [검증 결과](#검증-결과) 참고).
> 데이터 수집·통계 분석·전략 검증을 실습하는 용도로 쓰시고, 추천 번호는 참고용으로만
> 봐주세요.

## 설치

```bash
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
```

## 사용법

```bash
python main.py update              # 당첨번호 수집 (없는 회차만 증분 수집)
python main.py stats               # 통계 요약
python main.py combos              # 2개/3개 번호 조합 출현 분석
python main.py prize               # 전략별 '당첨 시 기대 상금' 비교
python main.py predict -n 5        # 다음 회차 번호 5게임 추천
python main.py backtest -d 200     # 최근 200회차로 전략 성능 검증
```

주요 옵션:

| 명령 | 옵션 | 설명 |
| --- | --- | --- |
| `update` | `--force` | 캐시를 무시하고 전부 다시 수집 |
| `update` | `--start N` | N회차부터 수집 (기본 1) |
| `stats` | `--recent N` | 최근 N회차 기준 집계 (기본 100) |
| `predict` | `-s, --strategy` | 예측 전략 선택 (기본 `balanced`) |
| `predict` | `--seed N` | 난수 시드 고정 (같은 결과 재현) |
| `predict` | `--show-scores` | 번호별 점수와 근거 지표 표 출력 |
| `predict` | `--no-filter` | 조합 필터 비활성화 |
| `predict` | `--telegram` | 추천 번호를 텔레그램으로 전송 |
| `backtest` | `-s, --strategy` | 특정 전략만 검증 (생략 시 전체 비교) |
| `backtest` | `-d, --draws` | 검증할 최근 회차 수 (기본 100) |

첫 `update`는 1회차부터 전부 받아 약 1분 걸리고, 이후에는 새로 열린 회차만 가져옵니다.

## 데이터 수집 방식

동행복권 사이트가 개편되면서 회차별 결과 페이지(`/lt645/result`)는 당첨번호를 HTML에
직접 담지 않고 자바스크립트로 그립니다. 그래서 두 갈래로 긁습니다.

1. **`/lt645/result` HTML 파싱** — 회차 선택 드롭다운(`data-value`)은 서버에서 렌더링되므로,
   여기서 현재까지 추첨된 전체 회차 목록과 최신 회차를 얻습니다.
2. **`/lt645/selectPstLt645InfoNew.do`** — 페이지 스크립트가 결과 슬라이드를 채울 때 호출하는
   같은 사이트의 조회 주소입니다. 한 번에 10회차씩 내려주므로 회차마다 페이지를 여는
   것보다 요청 수가 1/10로 줄어듭니다(1233회차 = 약 124회 요청).

수집한 데이터는 `data/lotto_history.csv`에 캐시되며, 요청 사이에 0.3~0.7초 무작위 대기를
두고 50건마다 중간 저장하므로 중간에 끊겨도 받은 만큼은 남습니다.

수집 항목: 회차, 추첨일, 당첨번호 6개, 보너스번호, 1등 당첨자 수, 1등 1게임당 당첨금액,
회차 판매금액.

## 분석 지표 (`lotto/analyzer.py`)

- `frequency` / `weighted_frequency` — 번호별 출현 횟수. 가중 버전은 반감기(기본 100회차)를
  적용해 최근 흐름에 무게를 둡니다.
- `gaps` / `mean_gap` — 번호별 미출현 기간과 평균 출현 간격.
- `pair_matrix` — 번호쌍 동시 출현 횟수 (45×45).
- `carryover_rate` — 직전 회차 번호가 다음 회차에 다시 나오는 평균 개수.
- `sum_stats`, `odd_even_distribution`, `range_distribution` — 조합 필터의 기준이 되는 분포.

## 예측 전략 (`lotto/predictor.py`)

각 전략은 45개 번호에 점수를 매기고, 그 점수를 확률로 삼아 6개를 비복원 추출합니다.

| 전략 | 가정 |
| --- | --- |
| `uniform` | 균등 무작위. 다른 전략을 비교하는 기준선 |
| `hot` | 최근 자주 나온 번호가 계속 나온다 |
| `cold` | 최근 덜 나온 번호가 나올 차례다 |
| `overdue` | 평균 출현 간격보다 오래 안 나온 번호가 나올 차례다 |
| `pair` | 직전 회차 번호와 자주 함께 나온 번호가 유리하다 |
| `balanced` | `hot` 0.4 + `overdue` 0.35 + `pair` 0.25 |
| `clt` | 중심극한정리 — 조합 합계가 이론 평균 138에 가까울수록 유리하다 |
| `lstm` | LSTM 시계열 모델이 과거 32회차 흐름에서 다음 회차를 예측한다 (torch 필요) |
| `pairwise` | 2개 조합 확률 — 이미 뽑은 번호와 자주 함께 나온 번호를 이어서 뽑는다 |
| `unpopular` | 남들이 안 사는 조합 — 당첨 확률이 아니라 당첨 시 기대 상금을 높인다 |

### `clt` 전략의 이론

1~45 모집단의 평균은 μ=23, 분산은 σ²=(45²−1)/12 ≈ 168.67입니다. 비복원 추출 6개
합계의 이론 분포는

- 평균 = 6μ = **138**
- 분산 = 6σ² · (N−n)/(N−1) = 6·168.67·(39/44) ≈ 897 (표준편차 ≈ **29.9**)

이고 중심극한정리에 따라 근사적으로 정규분포를 따릅니다. 이건 개별 번호가 아니라
조합 전체의 성질이라, `clt`는 번호별 가중치 대신 **조합 단위 기각 샘플링**으로
구현했습니다: 균등하게 6개를 뽑은 뒤 합계의 정규 밀도(최댓값 1로 정규화)를 채택
확률로 사용합니다. 그 결과 추천 조합의 합계가 138 주변에 집중됩니다(실측: 균등 추출
표준편차 29.9 → clt 21.3).

주의할 점: 실제 당첨 조합의 합계도 애초에 같은 분포를 따릅니다. "합계가 138 근처인
조합"이 자주 당첨되는 건 그런 조합의 **개수가 많아서**이지, 개별 조합이 유리해서가
아닙니다. 백테스트도 이를 그대로 보여줍니다.

### `lstm` 전략의 구조와 결과

과거 32회차의 당첨번호를 각각 45차원 멀티-핫 벡터로 인코딩한 시퀀스를 LSTM(은닉 96,
2층)에 넣고, 다음 회차에 각 번호가 포함될 확률 45개를 출력하도록 학습합니다(번호별
이진 분류, BCE 손실). 출력 확률을 번호별 가중치로 사용합니다. 백테스트에서는 50회차마다
재학습하며, 캐시된 모델이 현재 시점보다 미래 데이터로 학습된 경우 강제 재학습해 미래
정보 누출을 차단합니다.

torch가 필요한 선택 기능입니다:

```bash
pip install -r requirements-lstm.txt
python main.py predict -s lstm
```

**학습 결과가 곧 검증 결과입니다.** 1233회차 전체로 학습한 모델의 출력은 평균 0.1333 —
정확히 `6/45`입니다. 즉 모델이 시계열 패턴을 찾은 게 아니라 "모든 번호는 매 회차 6/45
확률로 나온다"라는 기저 출현율만 학습했습니다. 남은 ±0.02 수준의 편차는 전체 기간
출현 빈도를 약하게 반영한 것으로, 사실상 `hot` 전략의 복잡한 버전입니다. 추첨이 진짜
무작위라면 이것이 모델이 도달할 수 있는 최선이며, 백테스트에서도 무작위와 구별되지
않습니다(아래 표).

### `pairwise` 전략과 2개/3개 조합 분석

`pairwise`는 번호를 독립적으로 뽑지 않습니다. 첫 번호를 뽑은 뒤, 다음 번호는 *이미
뽑은 번호들과 과거에 함께 나온 횟수의 곱*에 비례해 순차적으로 선택합니다(라플라스
평활 α=1, 로그 공간 계산). 2개 조합의 경험적 확률을 조합 구성에 직접 사용하는
방식입니다.

조합 자체의 분포는 `combos` 명령으로 확인할 수 있습니다:

```bash
python main.py combos            # 2개/3개 조합 상위 출현 + 균등성 검정
python main.py combos --poisson  # 출현 횟수 분포를 포아송 이론값과 비교
```

1233회차 실측 결과:

| | 2개 조합 | 3개 조합 |
| --- | --- | --- |
| 조합 수 | 990 | 14,190 |
| 조합당 기대 출현 | 18.68회 | 1.74회 |
| 최다 출현 | 11-21 (34회) | 3-20-44 등 6종 (8회) |
| 카이제곱 z-점수 | −1.26 | −2.42 |
| 분산/평균 (포아송이면 ≈1) | 0.943 | 0.971 |

두 경우 모두 **균등 가설과 부합**합니다. "11-21이 34회 나온 것"도 990개 조합의
최댓값으로는 포아송 분포가 예측하는 정확히 그 수준입니다(출현 횟수 분포가 이론값과
거의 일치 — `--poisson`으로 확인). 즉 특별히 궁합이 좋은 조합은 존재하지 않으며,
pairwise가 학습하는 "함께 나온 빈도"는 전부 표본 잡음입니다.

### `unpopular` 전략 — 유일하게 통계적으로 유의한 우위

번호 예측 전략들이 전부 무작위와 구별되지 않는 것과 달리, 이 전략의 근거는 추첨기가
아니라 **구매자들의 행동**이라 실재합니다. 로또 1등 상금은 당첨자끼리 나누는 패리뮤추얼
방식이므로, 남들이 많이 사는 조합은 당첨돼도 상금이 쪼개집니다. 회차별 1등 당첨자 수를
"그 조합을 몇 명이 샀는가"의 관측치로 삼아 포아송 회귀를 적합합니다:

```
당첨자수 ~ Poisson(판매량 × exp(β·조합특성))     # python main.py prize 로 확인
```

1233회차 적합 결과 (z가 클수록 유의):

| 특성 | 계수 | z | 해석 |
| --- | --- | --- | --- |
| 합계 | −0.0018 | −2.0 | 큰 수 조합일수록 사는 사람이 적다 |
| 연속 번호 포함 | −0.0784 | **−4.0** | 사람들이 연속 번호를 기피한다 |
| 월 범위(1~12) 개수 | +0.0287 | +1.6 | 생일·월 숫자는 겹치기 쉽다 |

전략은 인기도 η의 역수 방향으로 조합을 뽑습니다(샘플링 ∝ exp(−γη), γ=4). 결과 검증
(`python main.py prize`, 최근 52회차 평균 판매량·상금 기준):

| 전략 | 예상 동시 당첨자 | 당첨 시 기대 상금 |
| --- | --- | --- |
| unpopular | 12.15명 | **24.0억** |
| uniform | 12.37명 | 23.6억 |

**당첨 확률은 그대로**이고(모든 조합 동일), 당첨됐을 때 나눠 갖는 사람이 줄어 기대
상금이 약 2% 높아집니다. 이것이 로또에서 통계적으로 방어 가능한 유일한 개선이며,
전체 기대수익률(세전 약 50%)은 여전히 음수라는 점은 변하지 않습니다.

뽑은 조합은 마지막에 **조합 필터**를 통과해야 합니다: 합계가 과거 분포의 5~95% 구간
이내, 홀짝이 한쪽으로 완전히 쏠리지 않을 것, 연속 숫자 4개 이상 금지, 같은 십의 자리
5개 이상 금지. 통계적으로 극단적인 조합을 걸러내는 장치입니다.

## 텔레그램으로 받기

추천 번호를 텔레그램 메시지로 보낼 수 있습니다.

```bash
export TELEGRAM_BOT_TOKEN='봇토큰'
export TELEGRAM_CHAT_ID='채팅ID'
python main.py predict -s unpopular -n 5 --telegram
```

봇 준비:

1. 텔레그램에서 [@BotFather](https://t.me/BotFather)에게 `/newbot` — 토큰을 받습니다.
2. 만든 봇과 대화를 시작합니다(먼저 아무 메시지나 보내야 봇이 답할 수 있습니다).
3. `https://api.telegram.org/bot<토큰>/getUpdates` 를 열어 `chat.id`를 확인합니다.

토큰과 채팅 ID는 **환경변수로만** 읽습니다. 코드나 저장소에 넣지 마세요(`.env`는
`.gitignore`에 포함돼 있습니다). 전송 실패 시 종료 코드 1과 함께 원인을 출력하며,
오류 메시지에 토큰이 섞이지 않도록 처리했습니다.

매주 토요일 추첨 전에 자동으로 받고 싶다면 cron에 등록할 수 있습니다:

```bash
# 매주 토요일 오후 6시 (crontab -e)
0 18 * * 6 cd /path/to/lotto_claude && .venv/bin/python main.py update && .venv/bin/python main.py predict -s unpopular -n 5 --telegram
```

## 검증 결과

백테스트는 과거 각 시점으로 돌아가 "그때까지의 데이터만" 가지고 번호를 뽑았다면 실제
당첨번호와 몇 개나 맞았을지 계산합니다(워크포워드 방식, 미래 정보 누출 없음).

최근 200회차 × 10게임 기준:

| 전략 | 평균 적중 (seed 42) | 평균 적중 (seed 7) |
| --- | --- | --- |
| lstm | 0.8340 | 0.7815 |
| pairwise | 0.7815 | 0.8080 |
| clt | 0.8240 | 0.7940 |
| balanced | 0.8195 | 0.8095 |
| pair | 0.8175 | 0.8060 |
| overdue | 0.8065 | 0.7870 |
| uniform (무작위) | 0.7970 | 0.7935 |
| hot | 0.7890 | 0.7980 |
| cold | 0.7875 | 0.7660 |

무작위 선택의 이론적 기댓값은 `6 × 6/45 = 0.8`개입니다. 모든 전략이 이 값 근처에
몰려 있고, 시드만 바꿔도 순위가 뒤집힙니다(lstm 0.8340 → 0.7815, clt 1위 → 4위).
전략 간 차이는 표본 오차 범위 안이며, **어떤 전략도 무작위보다 낫지 않습니다.**
이 프로젝트에서 얻을 수 있는 진짜 결론입니다.

```bash
python main.py backtest -d 200 -n 10 --seed 7   # 시드를 바꿔 순위 변동 확인
```
